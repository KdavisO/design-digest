# Verification Before Completion

## トリガー

以下の表現を使う **前** にこのスキルを適用する:

- 「完了しました」「修正しました」「テストがパスしました」
- 「動作確認できました」「問題ありません」
- 「実装が完了しました」「対応しました」

## 原則

**証拠なしの完了宣言は行わない。**

「おそらく動くはず」「パスするはず」という推測ではなく、実際にコマンドを実行してその出力を確認してから完了を宣言する。

## 検証チェックリスト

完了を宣言する前に、以下を **実際に実行** して結果を確認する:

### 1. テストの実行

```bash
# テストランナーを実行し、出力を確認する
# 「テストを書いたのでパスするはず」ではなく、実行結果を見る
```

- テストがすべてパスしていることを確認
- 新規テストが含まれている場合、そのテストが実際に実行されていることを確認（テストファイルが認識されていない場合がある）

### 2. ビルド・型チェック

```bash
# TypeScript の場合
# tsc --noEmit で型エラーがないことを確認
```

- コンパイルエラー・型エラーがないことを確認
- 警告がある場合、意図的なものか確認

### 3. lint

```bash
# lint を実行し、エラーがないことを確認
```

### 4. 変更内容の最終確認

```bash
git diff --staged   # コミット前: ステージ済みの差分を確認
git diff origin/main...HEAD  # PR 前: ブランチ全体の差分を確認
```

- 意図しない変更が含まれていないか
- デバッグ用コードが残っていないか
- ファイルの追加・削除が意図通りか

## 禁止表現

以下の表現は、検証なしの完了宣言を示唆するため使用しない:

- 「should pass」「should work」「おそらく動きます」
- 「great!」「done!」「完了!」（検証結果の提示なしで）
- 「probably works」「問題ないと思います」

代わりに:

- 「テストを実行し、全 N 件がパスしました」
- 「tsc --noEmit を実行し、型エラーがないことを確認しました」
- 「git diff で変更内容を確認し、意図通りの差分であることを確認しました」

## `.claude/rules/git-conventions.md` との関係

- セルフレビューチェックリストの各項目は、このスキルの検証結果に基づいて記入する
- 「確認しました」と記入する場合、実際にコマンドを実行した証拠があること
- このスキルはセルフレビューの **品質を担保する** 補完的な位置づけ
