# Brainstorming（設計ブレスト）

## トリガー

以下の場合にこのスキルを適用する:

- 新機能の設計方針が未確定のとき
- 複数の実装アプローチが考えられ、トレードオフの整理が必要なとき
- Issue の要件が曖昧で、実装前に設計の明確化が必要なとき

## ワークフロー

### 1. 要件の明確化（ヒアリング）

ユーザーに以下の観点で質問し、要件を明確化する:

- **目的**: この機能で解決したい課題は何か
- **スコープ**: 今回実装する範囲と、対象外とする範囲はどこか
- **制約**: 技術的制約、パフォーマンス要件、既存コードとの互換性
- **ユーザー体験**: 誰がどのように使うか、期待する振る舞いは何か

質問は一度にまとめて行い、ユーザーの回答を待つ。回答が十分であれば次のステップに進む。不足があれば追加質問する。

### 2. アプローチの提案（2〜3案）

要件を踏まえ、2〜3 の実装アプローチを提案する。各アプローチには以下を含める:

| 項目 | 内容 |
| --- | --- |
| アプローチ名 | 簡潔な名前（例: 「API Gateway パターン」） |
| 概要 | 1〜2 文での説明 |
| メリット | 箇条書きで 2〜3 点 |
| デメリット | 箇条書きで 2〜3 点 |
| 影響範囲 | 変更が必要なファイル・モジュールの概要 |
| 実装コスト | 小・中・大（相対的な見積もり） |

### 3. トレードオフ分析

提案したアプローチを以下の軸で比較する:

- **保守性**: 将来の変更に対する耐性
- **複雑性**: 実装・理解の難易度
- **パフォーマンス**: 実行時の効率
- **既存コードとの整合性**: 現在のアーキテクチャとの親和性

比較結果に基づき、推奨アプローチとその理由を提示する。最終的な選択はユーザーに委ねる。

### 4. 設計ドキュメントの作成

ユーザーがアプローチを選択した後、`docs/specs/` に設計ドキュメントを作成する:

- ファイル名: `docs/specs/{機能名}.md`
- テンプレート: `docs/specs/_template.md` に従う
- ブレストで決定した内容（選択したアプローチ、トレードオフの判断理由）を「概要」「技術的制約」セクションに反映する
- 不採用アプローチの概要と不採用理由は「技術的制約」セクションに簡潔に記録する（将来の設計見直し時の参考として）

### 5. 実装への接続

設計ドキュメント作成後、実装フェーズへの接続を行う:

- 設計ドキュメントのパスを提示し、実装時の参照先を明確にする
- 必要に応じて TDD スキル（`/test-driven-development`）との連携を提案する

## 既存フローとの関係

### `docs/specs/_template.md` との統合

- 設計ドキュメントは `_template.md` のフォーマットに従う
- ブレストの成果物がテンプレートの各セクション（概要、要件、技術的制約、受け入れ基準）に直接マッピングされる

### `/test-driven-development` との連携

| フェーズ | 担当スキル |
| --- | --- |
| 設計（何を作るか、どう作るか） | brainstorming |
| 実装（テストファーストで作る） | test-driven-development |

brainstorming で設計を固めた後、test-driven-development でテストファーストの実装に移行する。

### `.claude/rules/git-conventions.md` との関係

- 設計ドキュメントの作成は独立したコミットとする（コミットプレフィックス: `docs:`）
- 実装コミットとは分離し、レビュー時に設計意図を追跡可能にする
