# Systematic Debugging

## 基本原則

**根本原因を特定せずに修正しない。**

症状だけを見て「とりあえず動くように直す」のではなく、なぜその不具合が発生しているのかを理解してから修正する。

## トリガー

以下の場合にこのスキルを適用する:

- バグ報告・不具合の調査（`fix:` コミット）
- テスト失敗の原因調査
- 予期しない動作の原因特定
- 本番環境で発生した障害の分析

## ワークフロー: 4フェーズ根本原因調査

### フェーズ1: 症状の収集と再現

問題を正確に理解し、再現可能な状態にする。

1. **症状の記録**: エラーメッセージ、スタックトレース、ログ出力、再現手順を収集する
2. **再現の確認**: 症状を確実に再現できるテストまたは手順を確立する
3. **影響範囲の特定**: 問題が発生する条件（入力値、環境、タイミング）と発生しない条件を切り分ける

```
# NG: 「エラーが出る」だけで次に進む
# OK: 再現手順を確立し、どの条件で発生するか特定してから次に進む
```

**チェックポイント**: 問題を確実に再現できるか。再現できない場合はログ追加や条件の絞り込みを行い、再現可能にしてからフェーズ2に進む。

### フェーズ2: 仮説の立案

収集した症状から、根本原因の候補を列挙する。

1. **仮説を複数立てる**: 少なくとも2つ以上の仮説を立案する（最初の仮説に飛びつかない）
2. **仮説に優先順位を付ける**: 症状との整合性、発生確率、検証の容易さを基準に優先順位を決める
3. **検証方法を定義する**: 各仮説に対して「この仮説が正しければ○○が観測される」という検証可能な予測を記述する

仮説の立案にあたっては以下の観点を検討する:

- データフロー: 入力から出力までの経路で、どこでデータが変質しているか
- 状態管理: 想定外の状態遷移や競合状態が起きていないか
- 境界条件: null/undefined、空配列、0、負数、型の不一致が関与していないか
- 外部依存: API レスポンス、DB 状態、ファイルシステム、環境変数に起因していないか
- タイミング: 非同期処理の順序、タイムアウト、リトライに起因していないか

### フェーズ3: 仮説の検証（根本原因の特定）

仮説を1つずつ検証し、根本原因を特定する。

1. **最も可能性の高い仮説から検証する**: 優先順位に従い、1つずつ検証する
2. **検証は観察で行う**: コードリーディング、ログ追加、デバッグ出力、テストケース追加で仮説を検証する
3. **仮説が棄却されたら次へ**: 検証結果が仮説と矛盾する場合、その仮説を棄却して次の仮説に進む
4. **根本原因を確定する**: 「この原因を取り除けば症状が消える」と確信できる根拠を得る

```
# NG: 「たぶんここが原因」で修正を始める
# OK: 「この変数がこの時点で null になっている」という具体的な証拠を得る
```

**チェックポイント**: 根本原因を1文で説明できるか。できない場合は検証が不十分。

### フェーズ4: 修正と回帰テスト

根本原因に対する修正を行い、再発を防止する。

1. **根本原因に対する修正**: 症状の回避策ではなく、根本原因を解消するコードを書く
2. **回帰テストの作成**: フェーズ1で確立した再現手順をテストケースとして実装する（`/test-driven-development` と連携）
3. **修正の検証**: テストがパスし、元の症状が解消されたことを確認する（`/verification-before-completion` と連携）
4. **副作用の確認**: 修正が他の機能に影響していないことを確認する（既存テストの全パス）

## 既存スキルとの連携

| フェーズ | 連携スキル | 連携内容 |
| --- | --- | --- |
| フェーズ1（症状収集） | — | 独立して実施 |
| フェーズ2（仮説立案） | — | 独立して実施 |
| フェーズ3（仮説検証） | — | 独立して実施 |
| フェーズ4（修正） | `/test-driven-development` | 回帰テストを TDD の RED フェーズとして作成し、修正を GREEN フェーズとして実装する |
| フェーズ4（検証） | `/verification-before-completion` | 修正完了の宣言前に、テスト実行・ビルド・lint の実行証拠を確認する |

### TDD スキルとの使い分け

| 状況 | 使用するスキル |
| --- | --- |
| 新機能の実装 | `/test-driven-development` のみ |
| 原因が明白な軽微なバグ | `/test-driven-development` のみ（RED で再現テスト → GREEN で修正） |
| 原因が不明なバグ・複雑な不具合 | `/systematic-debugging`（フェーズ1〜3で原因特定 → フェーズ4で TDD と連携） |

## アンチパターン

- **推測で修正する**: 仮説を検証せずに「たぶんここが原因」で修正を始めること
- **症状だけを消す**: 根本原因を残したまま、エラーを握りつぶしたり条件分岐で回避すること
- **1つの仮説に固執する**: 最初の仮説が棄却されても他の可能性を検討しないこと
- **検証なしで完了する**: 修正後にテストを実行せずに「直しました」と宣言すること
- **再現なしで調査する**: 問題を再現できないまま原因を推測し続けること

## `.claude/rules/git-conventions.md` との関係

- デバッグで追加したログやデバッグコードは、修正コミットに含めない（セルフレビューの「デバッグコード削除」で確認）
- 回帰テストと修正コードは別コミットに分ける（git-conventions.md の「本体コード vs テストコード」規約に従う）
- コミットメッセージには根本原因の要約を含める（例: `fix: null チェック漏れによる子プロフィール表示エラーを修正`）
