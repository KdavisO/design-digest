# Web 調査委譲ガイドライン

## 概要

実装・エラー対応・設計判断の過程で、Claude Code 自身の知識では不十分と判断した場合に外部の Web 検索手段を利用する。検索の規模と推論の必要性に応じて **Brave Search MCP**（軽量検索）、**o3-search-mcp**（推論付き検索）、**gemini-analyzer エージェント**（大規模調査）を使い分ける。調査モード（`research` ラベル）以外の通常の実装フローでも適用される。

## 検索手段の使い分け

| 用途 | 手段 | 例 | コスト目安 |
| --- | --- | --- | --- |
| 軽い検索（単発の事実確認） | **Brave Search MCP** | エラーメッセージの検索、API仕様の確認、最新バージョン確認 | 無料（月2,000クエリ） |
| 推論付き検索（分析が必要な検索） | **o3-search-mcp** | 技術比較調査、エラーの根本原因分析、設計判断の裏付け | 約$0.05〜$0.15/回 |
| 大規模調査（網羅的スキャン） | **gemini-analyzer** | 複数観点の調査、ベストプラクティスの網羅的収集 | エージェント起動コスト |

### Brave Search MCP を使う場合

- エラーメッセージやスタックトレースの検索
- 特定の API・関数・設定値の仕様確認
- ライブラリの最新バージョンやリリースノートの確認
- 単発の事実確認（「X は Y をサポートしているか？」等）

### o3-search-mcp を使う場合

- 技術的な比較調査（「A と B のどちらが適切か？」等）
- エラーの根本原因分析（単純検索では解決しない複雑なエラー）
- 設計判断に推論が必要な場合（トレードオフの分析、アーキテクチャ選定等）
- 検索結果に対する要約・分析が必要な場合

### gemini-analyzer に委譲する場合

- 複数の観点やソースを横断する網羅的な調査
- 設計判断のための外部事例・ベストプラクティスの収集
- 調査モード（`research` ラベル）での包括的な情報収集

## 判断基準

以下の **いずれか** に該当する場合、外部の Web 検索を利用する（上記の使い分けに従い、適切な手段を選択する）:

1. **知識カットオフ後の情報が必要**: Claude Code の学習データのカットオフ以降にリリース・更新されたライブラリ・ツール・API に関する情報が必要な場合（バージョン互換性、破壊的変更、新 API 仕様など）
2. **エラー解決が行き詰まった場合**: エラーメッセージやスタックトレースに対して自身の知識で解決策が見つからず、かつ自動修正の試行（2回程度）でも解決しない場合
3. **外部事例の参照が有効な場合**: 設計判断で複数のアプローチが考えられ、外部のベストプラクティスや採用事例の参照が判断の質を向上させると見込まれる場合

## 検索しない場合

以下に該当する場合は外部検索を **しない**（オーバーヘッドが利益を上回る）:

- 自身の知識で十分に回答・解決できる場合
- プロジェクト固有のコードや設定の問題で、外部情報が役立たない場合
- 単純な構文エラーやタイポなど、原因が明白な場合
- 既にプロジェクト内のドキュメント（`docs/`、`README.md` 等）に答えがある場合

## Brave Search MCP の利用手順

Brave Search MCP は `.mcp.json` で設定済みのため、Claude Code セッション内で直接ツールを呼び出せる。

1. `brave_web_search` ツールを使用して検索クエリを実行する
2. 検索結果から必要な情報を抽出し、実装・修正に反映する

```text
# 使用例
brave_web_search で "Next.js middleware redirect loop" を検索
```

※ `BRAVE_API_KEY` 環境変数が設定されている必要がある（設定手順は `SETUP.md` を参照）

## o3-search-mcp の利用手順

o3-search-mcp は `.mcp.json` で設定済みのため、Claude Code セッション内で直接ツールを呼び出せる。OpenAI o3 の推論能力を活用し、検索結果に対する分析・要約を含むレスポンスを返す。

1. `web_search` ツールを使用して検索クエリを実行する
2. 推論付きの検索結果から必要な情報を抽出し、実装・修正に反映する

```text
# 使用例
web_search で "React Server Components vs Client Components トレードオフ" を検索
```

※ `OPENAI_API_KEY` 環境変数が設定されている必要がある（Tier 4 以上。設定手順は `SETUP.md` を参照）
※ コスト: 約 $0.05〜$0.15/回（`REASONING_EFFORT` 設定による）

## gemini-analyzer への委譲手順

大規模調査が必要な場合は gemini-analyzer エージェントに委譲する:

1. gemini-analyzer エージェントを `subagent_type: "gemini-analyzer"` で起動する
2. 用途に応じた指示テンプレート（`gemini-analyzer.md` の「用途別テンプレート」セクション参照）を使用し、調査テーマと観点を具体的に指定する
3. 調査結果を受け取り、Claude Code 自身がプロジェクトへの適用可否を判断する
4. 調査結果に基づいて実装・修正を続行する

## 並列エージェントへの適用

`/parallel-suggest` 経由で起動されたチームメイト（worktree エージェント）にも同じ判断基準が適用される。各チームメイトは自身の実装中に上記の判断基準に基づき、独立して gemini-analyzer を呼び出してよい。

## 注意事項

- gemini-analyzer の起動にはオーバーヘッド（数十秒〜数分）があるため、頻繁な呼び出しは避ける
- 1つの実装タスクにつき、Web 調査の委譲は原則 **2回以内** に留める
- 調査結果の URL は検証可能な形で保持し、実装の根拠として参照できるようにする
