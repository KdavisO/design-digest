# Writing Plans（実装計画作成）

## トリガー

以下の場合にこのスキルを適用する:

- 設計ブレスト（`/brainstorming`）の完了後、実装に着手する前
- Issue の変更対象が複数ファイル・複数モジュールにまたがる場合
- 実装の順序や依存関係を整理する必要がある場合

## ワークフロー

### 1. 設計情報の確認

`/brainstorming` で作成された設計ドキュメント（`docs/specs/{機能名}.md`）を参照し、以下を確認する:

- 選択されたアプローチの概要
- 技術的制約
- 受け入れ基準

設計ドキュメントが存在しない場合（小規模タスクや Issue の要件が明確な場合）は、Issue の要件から直接タスク分解に進む。

### 2. タスク分解

実装を論理的な単位に分解する。各タスクには以下の情報を含める:

| 項目 | 内容 |
| --- | --- |
| タスク名 | 簡潔な名前（例: 「認証ミドルウェアの追加」） |
| 対象ファイル | 変更・作成するファイルパスの一覧 |
| 変更内容 | 具体的に何をするか（関数追加、型定義変更、設定変更等） |
| テスト方法 | このタスクの正しさをどう検証するか |
| コミットプレフィックス | `feat:`, `test:`, `refactor:` 等 |

### 3. 実装順序と依存関係の整理

タスクを実装順序に並べ、依存関係を明示する:

- **依存関係**: タスク B はタスク A の完了後に着手（例: 型定義 → 実装 → テスト）
- **並列可能**: タスク C と D は独立しており、順序を問わない
- **コミット粒度**: `.claude/rules/git-conventions.md` の「1つの変更目的 = 1コミット」に対応させる

### 4. YAGNI・DRY の適用

タスク分解後、各タスクについて以下を確認する:

- **YAGNI**: 現在の要件に不要な機能やインターフェースが含まれていないか
- **DRY**: 既存コードとの重複がないか。既存の関数・モジュールで代替可能なものはないか
- **過剰な抽象化**: 3つ未満の類似ケースに対して抽象化を導入していないか

不要と判断したタスクは計画から除外し、除外理由を記録する。

### 5. 計画の提示

タスク分解の結果を以下の形式でユーザーに提示する:

```
=== 実装計画 ===
設計ドキュメント: docs/specs/{機能名}.md（または「なし」）

タスク1: {タスク名}
  対象: {ファイルパス1}, {ファイルパス2}
  内容: {変更内容}
  テスト: {テスト方法}
  コミット: {プレフィックス} {コミットメッセージ案}

タスク2: {タスク名}
  依存: タスク1
  ...

除外したタスク:
  - {タスク名}: {除外理由}（YAGNI/DRY/既存コードで代替可能）
================
```

ユーザーの承認後、実装に着手する。

## 既存スキルとの連携

### `/brainstorming` との連携

| フェーズ | 担当スキル |
| --- | --- |
| 設計（何を作るか、どう作るか） | `/brainstorming` |
| 計画（どう分解して実装するか） | `/writing-plans` |
| 実装（テストファーストで作る） | `/test-driven-development` |

brainstorming が設計ドキュメントを作成した後、writing-plans がそのドキュメントを入力としてタスク分解を行う。brainstorming が不要な場合（要件が明確な小規模タスク）は、writing-plans から直接開始できる。

### `/test-driven-development` との連携

- 計画の各タスクに「テスト方法」を明記することで、TDD の RED フェーズで書くべきテストケースが事前に特定される
- テストタスクと実装タスクを別コミットとして計画する（`.claude/rules/git-conventions.md` の「本体コード vs テストコード」規約に従う）

### `docs/specs/_template.md` との関係

- 設計ドキュメントの「ファイル配置」セクションは概要レベルの情報を提供する
- writing-plans はそれをさらに詳細化し、各ファイルに対する具体的な変更内容と実装順序を定義する
- 設計ドキュメントが「何を作るか」を定義し、実装計画が「どの順序でどう作るか」を定義する

## `.claude/rules/git-conventions.md` との関係

- 各タスクは `.claude/rules/git-conventions.md` のコミット粒度ガイドラインに対応させる
- 「1つの変更目的 = 1コミット」をタスク分解の粒度基準とする
- 新規ファイル追加と既存ファイル修正は別タスク（= 別コミット）として計画する
- 計画段階でコミットプレフィックスを決定し、セルフレビュー時のコミットメッセージ確認を効率化する
