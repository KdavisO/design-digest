# 大規模Figmaファイル取得の調査結果

## 調査日: 2026-03-26

## A. REST API の調査結果

### レスポンスサイズ制限

- **公式の明示的なサイズ制限はドキュメントに記載なし**
- **55秒のタイムアウト制限**: `GET /v1/files/:key` と `GET /v1/images/:key` に適用
- Cloudflare インフラによる実質的な制限あり（報告ベースで ~280-320MB 程度）

### 大規模ファイル取得の公式推奨方法

Figma REST API v1 には**ページネーション機能は存在しない**。レスポンスサイズを制御する手段は以下の2つのみ:

1. **`depth` パラメータ**: ノードツリーの走査深度を制限
2. **`ids` パラメータ** (`/nodes` エンドポイント): 特定ノードのみ取得

推奨戦略: **"metadata-first, prune-first, fetch-last"**
1. `depth=1` でページレベルのノードIDを取得
2. ローカルでフィルタリング
3. 必要なノードのみ個別に取得

### バッチサイズ

- `ids` パラメータの公式な最大数制限はドキュメントに記載なし
- コミュニティの実践: ~500 ノードID/リクエスト（2-10秒のディレイ付き）

### レスポンスサイズに影響するパラメータ

| パラメータ | 効果 | 推奨 |
|-----------|------|------|
| `geometry=paths` | ベクターデータを含む → サイズ大幅増加 | 不要なら省略 |
| `plugin_data` | プラグインデータを含む | 不要なら省略 |
| `branch_data` | ブランチメタデータを含む | 不要なら省略 |

### レートリミット

- プランに応じて 10-20 リクエスト/分（Dev/Full シート）
- 429レスポンス時の `Retry-After` ヘッダー対応は実装済み

## B. Figma MCP / Plugin API の調査結果

### Figma MCP サーバーの概要

- **Remote MCP サーバー** (推奨): `https://mcp.figma.com/mcp` に接続
- **Desktop MCP サーバー**: ローカルのFigmaデスクトップアプリ経由

### DesignDigest での利用可否

**結論: 現時点では利用不可**

| 要件 | 状況 |
|------|------|
| ヘッドレス環境（GitHub Actions）での動作 | ❌ 非対応。OAuth認証フローがインタラクティブ |
| Personal Access Token 認証 | ❌ 公式MCPサーバーは非対応 |
| CI/CD パイプラインでの自動実行 | ❌ コードエディタ連携前提の設計 |
| ペイロードサイズ制限の回避 | ❓ 未確認。MCP も同じレートリミットを共有 |

### `use_figma` ツールの内部実装

- 内部的に REST API / Plugin API のどちらを使用しているかは**非公開**
- Desktop サーバーの書き込み操作は WebSocket → Desktop Bridge Plugin → Plugin API
- Remote サーバーは内部 API を使用している可能性が高い

### サードパーティ MCP サーバー

REST API ベースのサードパーティ MCP サーバー（例: `thirdstrandstudio/mcp-figma`）は Personal Access Token で動作するため CI 環境でも利用可能だが、公式 MCP の `use_figma` 機能は使えない。

## C. 改善方針の決定

### 採用方針: **A. REST API の最適化**（現行アプローチの改善）

理由:
1. Figma MCP はヘッドレス環境で動作しないため、GitHub Actions ベースの DesignDigest では利用不可
2. REST API の最適化により、現在のエラー駆動フォールバックを事前推定ベースに改善可能
3. API コール数を削減可能

### 具体的な改善内容

1. **事前推定ベースのチャンク分割**: エラー発生を待たず、`depth=1` でのノード数から事前にチャンク分割の必要性を判断
2. **適応的バッチサイズ**: 子ノード数に応じてバッチサイズを動的に調整
3. **`FIGMA_BATCH_SIZE` 設定の追加**: ユーザーが環境に応じてバッチサイズを調整可能に
4. **エラーフォールバックの維持**: 事前推定で漏れた場合の安全網として現行のフォールバックも維持

### API コール数の比較

**現行方式**（エラー駆動フォールバック）:
- 正常時: 1 API コール（全体取得）
- エラー時: 1（失敗） + 1（depth=1取得） + ceil(N/5)（チャンク取得）= 2 + ceil(N/5)

**改善後**（事前推定ベース / `fetchFileProactive`）:
- 共通: 常に最初に 1（`depth=1` でページメタデータ取得）
- 小規模ファイル: 1（depth=1取得） + 1（単一バッチでの `/nodes` 取得）= 最低 2 コール
  - ※ 従来の `GET /files/:key` 単体呼び出し（1コール）と比べて、メタデータ取得分だけ +1 コールとなる
- 大規模ファイル: 1（depth=1取得） + ceil(N/batchSize)（チャンク取得）
  - `precomputedShallow` により `fetchNodesChunked` 内の冗長な depth=1 discovery を省略
  - 失敗する全体取得（1コール）を前提としたリトライを行わないため、タイムアウトリスクと無駄な全体取得コールを削減
  - 小規模時の余分な 1 コールと引き換えに、大規模時の安定性を向上させるトレードオフ
