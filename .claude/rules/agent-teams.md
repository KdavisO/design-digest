# Agent Teams ガイドライン

> **注意**: Agent Teams は実験的機能です。`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` の設定が必要です。動作が不安定な場合があります。

## Agent Teams とは

Agent Teams は Claude Code の実験的機能で、リード（メインセッション）が複数のチームメイトを動的に生成し、協調してタスクを遂行する仕組みです。各チームメイトは独立したコンテキストウィンドウを持ち、共有タスクリストを通じて自律的に作業を進めます。

チームメイトは定義ファイルではなく、自然言語のリクエストで動的に作成されます。

## サブエージェント vs Agent Teams の使い分け

| 基準 | サブエージェント（Agent ツール） | Agent Teams |
| --- | --- | --- |
| **タスク間の依存関係** | 独立したタスク（相互通信不要） | 相互依存のあるタスク（調整が必要） |
| **コンテキスト共有** | 不要（各自が完結） | 必要（観点の分担・結果の統合） |
| **通信方式** | 親エージェントに結果を返すのみ | ピアツーピアのメッセージング |
| **タスク管理** | 親が全てを管理 | 共有タスクリストで自律調整 |
| **典型的なユースケース** | 独立した Issue の並列実装（worktree） | レビュー観点分担、巡回分担、相互依存タスク |
| **worktree との併用** | ○（`isolation: "worktree"`） | △（同一ブランチでの協調作業向き） |
| **定義方法** | `.claude/agents/` ディレクトリ内のファイル（必要に応じて作成）、または inline の Agent 指示で都度起動 | 自然言語でリクエスト |
| **セッション跨ぎ** | 都度起動 | セッション内のみ（`/resume` 不可） |

### いつ Agent Teams を使うべきか

- **レビューコメントが多い場合（目安: 5件以上）**: 観点別（セキュリティ、パフォーマンス、テスト等）にチームメイトを分担
- **プロジェクト巡回**: コード品質、PR状態、ドキュメント整合性など対象ごとに分担
- **相互依存のあるタスクの並列実行**: API変更 + フロントエンド対応など、インターフェース調整が必要なケース
- **セルフレビューの強化**: 実装者とは別のチームメイトがレビュアー役を担当

### いつサブエージェント（従来方式）を使うべきか

- **完全に独立した Issue の並列実装**（現行の worktree パターン）
- **単純な調査・検索タスク**の並列実行
- **チームメイト間の通信が不要**なケース
- **セッションを跨いで作業が必要**なケース

## チームメイトの作成方法

チームメイトは自然言語でリクエストして動的に作成する:

```
# レビュー対応の例
レビューコメントが多いため、Agent Teams で対応します。
以下のチームメイトを作成してください:
- セキュリティレビュアー: セキュリティ関連の指摘を分析・対応
- コード品質レビュアー: リファクタリング・パフォーマンス指摘を対応
- テストレビュアー: テストカバレッジ・エッジケースの指摘を対応
```

### チームメイト作成時のベストプラクティス

- **役割は明確に分離する**: 各チームメイトの担当範囲が重複しないようにする
- **指示は具体的に**: 何を確認すべきか、どのような基準で判断すべきかを明記する
- **チームメイト数は最小限に**: 必要な観点の数に応じて 2〜4 名程度に抑える
- **モデルを指定可能**: 必要に応じて「各チームメイトにSonnetを使用」などと指定

## 表示モード

`.claude/settings.json` で表示モードを設定:

```json
{
  "teammateMode": "auto"
}
```

| モード | 説明 |
| --- | --- |
| `auto`（デフォルト） | tmux内ならスプリットペイン、それ以外はインプロセス |
| `in-process` | 全チームメイトがメインターミナルで実行（Shift+Down で切替） |
| `tmux` | 各チームメイトが独立したペインで実行 |

## 操作方法

- **Shift+Down**: チームメイトを切り替え（インプロセスモード）
- **直接メッセージ**: チームメイトのペインに移動してタイプ
- **Escape**: 現在のターンを中断
- **Ctrl+T**: タスクリストを表示

## 制約事項

- 1セッション1チーム（複数チームの同時実行不可）
- チームメイトはネストしたチームを作成不可
- セッション復元（`/resume`, `/rewind`）でチームメイトは復元されない
- リード（作成元セッション）は固定

## 既存コマンドでの Agent Teams 活用

### `/review-respond` での活用

レビューコメントが多い場合（目安: 5件以上）、`--team` フラグで Agent Teams モードを有効化できる。

- コメントを観点別に分類し、チームメイトに分担して対応
- 環境変数 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` が `"1"` でない場合（未設定を含む）はフォールバックとして従来の逐次処理を使用

### `/patrol` での活用

巡回対象が複数の場合、`--team` フラグで Agent Teams モードを有効化できる。

- 巡回対象（コード、PR、Issue、ドキュメント）ごとにチームメイトを作成し、並列分担で深い分析を実施
- 巡回対象が1つのみの場合はチームメイト作成のオーバーヘッドを避け、逐次処理を使用
- 環境変数 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` が `"1"` でない場合（未設定を含む）はフォールバックとして従来の逐次処理を使用

### `/parallel-suggest` での活用

推奨セット選択後、Agent Teams + worktree のハイブリッド方式で並列実行を開始する。

> **比較表との関係**: 比較表では「独立した Issue の並列実装」をサブエージェント + worktree の典型例としているが、`/parallel-suggest` では以下の理由から Agent Teams を採用する:
> - **tmux ペインによるリアルタイム進捗表示**: サブエージェント（バックグラウンド）では進捗が見えないが、Agent Teams の tmux 分割で各 Issue の実装状況をリアルタイムに確認できる
> - **承認プロンプトへの対応**: サブエージェントがバックグラウンドで承認待ちになるとスタックするが、Agent Teams ならペイン移動で即座に対応可能
> - 各チームメイトは独立した worktree で作業するため、ファイル競合は発生しない（サブエージェント + worktree と同等の分離性を確保）

- 各 Issue に対してチームメイトを作成し、それぞれが独立した worktree で実装・PR作成まで完結する
- `/patrol` や `/review-respond` と異なり `--team` フラグは不要。環境変数 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` が `"1"` に設定されている場合、セット選択後の実行がデフォルトで Agent Teams 方式となる（並列実行の開始がコマンドの主目的であるため）
- チームメイトの `subagent_type` は `general-purpose`（worktree でのファイル編集・Bash実行が必要なため）
- 環境変数 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` が `"1"` でない場合（未設定を含む）はフォールバックとして従来の `/issue-start XXX --parallel` コマンド一覧を表示
