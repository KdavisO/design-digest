# Git規約

## ブランチ命名

- フォーマット: `{type}/{issue番号}-{英語の短い説明}`
- type: `feat`, `fix`, `refactor`, `ui`, `docs`, `chore`
- 例: `feat/12-add-child-profile`, `fix/8-auth-redirect`

## ブランチ作成手順

新しいブランチを作成する前に、必ずmainを最新化する:

1. `git checkout main`
2. `git pull --ff-only origin main`
3. `git checkout -b {type}/{issue番号}-{英語の短い説明}`

※ worktree使用時は `git fetch origin main` の後に `git worktree add <path> -b {type}/{issue番号}-{英語の短い説明} origin/main` で最新のmainからブランチを作成する

## コミットメッセージ

- プレフィックス必須: `feat:`, `fix:`, `refactor:`, `ui:`, `docs:`, `chore:`, `test:`
- 日本語での記述可
- 例: `feat: 子どもプロフィール追加機能を実装`

## コードレビュー（セルフレビュー必須）

### 実施タイミング

- **すべてのコミット前にセルフレビューを必ず実施する**
- 自動コミット時も例外なく実施する（「自動コミットルール」参照）

### セルフレビュー手順

1. `git status` で未追跡ファイル・未ステージの変更を確認
2. `git add <対象ファイル>` でコミット対象をステージング
3. `git diff --staged` でコミット対象の差分を確認
4. 以下のチェックリストを順番に確認
5. 問題があれば修正してから再度レビュー（手順1からやり直し）
6. すべてOKならコミットを実行

### セルフレビューチェックリスト

- [ ] **可読性・保守性**: コードは読みやすく、意図が明確か
- [ ] **型安全性**: TypeScriptの型定義は適切か（`any`を避けているか）
- [ ] **セキュリティ**: APIキーの露出、RLS設定漏れ、要配慮個人情報の暗号化漏れはないか
- [ ] **既存コードとの整合性**: 命名規則、ディレクトリ構造、コーディングスタイルは統一されているか
- [ ] **デバッグコード削除**: 不要な`console.log`、コメントアウトしたコード、TODO残りはないか
- [ ] **コミットメッセージ**: 変更内容を適切に表しているか（プレフィックス付き）

### コミット前の出力フォーマット（必須）

**コミット実行前に必ず会話の中で以下の形式で出力すること：**
（Claude Codeがコミット前にユーザーに見える形で出力する）

```
セルフレビュー実施:
- [x] 可読性・保守性: （確認内容を記載）
- [x] 型安全性: （確認内容を記載、該当なしの場合は「N/A」）
- [x] セキュリティ: （確認内容を記載）
- [x] 既存コードとの整合性: （確認内容を記載）
- [x] デバッグコード削除: （確認内容を記載、該当なしの場合は「N/A」）
- [x] コミットメッセージ: （確認内容を記載）
```

この出力がない状態でのコミットは禁止。

## PR規約

- `Closes #XX` でIssueをリンクする
- 本文に概要・変更内容・テスト方法を含める
- assigneeに `{github_username}` を設定する
- reviewerを設定する（`gh api` を使用）

## 自動コミットルール

- 作業の論理的な区切りごとに、**ユーザー確認なしで**自動的にコミットする（ただしセルフレビューは必須）
- **自動コミット時もセルフレビューを必ず実施する**（上記「セルフレビューチェックリスト」「コミット前の出力フォーマット」参照）
- 以下の場合のみユーザーに確認を求める:
  - mainブランチへの直接コミット（これは禁止事項だが万が一の場合）
  - 既存コードの大幅な削除や書き換え（ファイル削除、50行以上の削除等）
- それ以外（設定ファイル変更、新規ファイル追加、通常の編集）は全て自動コミット

### 自動コミットの流れ

1. 変更作業を実施
2. `git status` で変更内容を確認
3. `git add <対象ファイル>` でコミット対象をステージング
4. `git diff --staged` でコミット対象の差分を確認
5. **セルフレビューチェックリストを確認**（必須）
6. 問題があれば修正してから再度レビュー（手順2からやり直し）
7. すべてOKならコミット実行

## コミット粒度ガイドライン

- 1つの変更目的 = 1コミット
- 以下は別コミットに分ける:
  - 新規ファイル追加 vs 既存ファイル修正
  - 機能実装 vs リファクタリング
  - 本体コード vs テストコード
- 関連する複数ファイルの変更は1コミットにまとめてOK（例: コンポーネント + 型定義）

## 禁止事項

- mainブランチへの直接コミット禁止（必ずブランチを切ってPR経由）
- `--force` push禁止
- `--no-verify` 禁止
