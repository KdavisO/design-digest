# Differential Review（セキュリティ差分レビュー）

Trail of Bits のセキュリティレビュー手法を参考にした差分レビュースキル。

## トリガー

以下の場合にこのスキルを適用する:

- セキュリティに関わるコード変更のレビュー時
- 認証・認可・暗号・外部API連携の変更を含むPR
- `/review-respond` でセキュリティ観点の深い分析が必要な場合

## ワークフロー

### Phase 0: トリアージ（変更の分類）

差分を取得し、各ファイルをリスクレベルで分類する。

```bash
# 変更ファイル一覧を取得
git diff --name-only origin/main...HEAD

# 差分の統計情報
git diff --stat origin/main...HEAD
```

**リスクレベルの判定基準:**

| リスク | トリガー条件 | 例 |
| --- | --- | --- |
| **HIGH** | 認証・暗号・外部呼び出し・入力バリデーション・権限チェック | auth/, middleware/, crypto, API key handling |
| **MEDIUM** | ビジネスロジック・状態変更・新規パブリックAPI・DB操作 | services/, models/, controllers/ |
| **LOW** | UI・コメント・テスト・ログ・設定ファイル | components/, *.test.*, *.config.* |

**HIGHリスクファイルから順に分析する。**

### Phase 1: コード分析

各ファイルの変更を以下の観点で分析する:

1. **入力バリデーション**: ユーザー入力がサニタイズ・検証されているか
2. **認証・認可**: 適切な認証チェックが存在するか、権限昇格の経路はないか
3. **暗号処理**: 安全なアルゴリズム・鍵長・モードが使用されているか
4. **外部呼び出し**: APIキー・シークレットの露出、SSRF、コマンドインジェクションのリスク
5. **エラーハンドリング**: エラーメッセージに機密情報が含まれていないか

### Phase 2: テストカバレッジ確認

セキュリティ関連の変更に対するテストが存在するか確認する:

- 認証バイパスのテスト
- 不正入力に対するバリデーションテスト
- 権限エラーのテスト
- エッジケース（null、空文字列、最大長超過）のテスト

### Phase 3: ブラスト半径の評価

変更が影響する範囲を評価する:

```bash
# 変更されたファイルを参照している箇所を検索
# （importやrequire元を確認）
```

- 直接の依存元（1ホップ）を特定
- パブリックAPIの変更がある場合、利用者への影響を評価
- セキュリティ境界（認証ミドルウェア等）の変更は影響範囲が広いため特に注意

### Phase 4: 攻撃者モデリング

HIGHリスクの変更に対して、攻撃者の視点で評価する:

- **攻撃者の目標**: この変更を悪用して何が達成できるか
- **攻撃経路**: どのような入力・操作で脆弱性をトリガーできるか
- **緩和策**: 既存の防御機構が有効か、追加の対策が必要か

### Phase 5: レポート出力

発見事項を以下の形式で報告する:

```
## セキュリティレビュー結果

### エグゼクティブサマリー
- レビュー対象: {PR/コミット}
- リスク評価: {HIGH/MEDIUM/LOW}
- 発見事項: {Critical: N件, Important: N件, Informational: N件}

### 発見事項

#### [Critical/Important/Informational] {タイトル}
- **場所**: {ファイル:行番号}
- **説明**: {脆弱性の説明}
- **影響**: {攻撃シナリオと影響範囲}
- **推奨修正**: {具体的な修正方法}

### 推奨事項
- {追加すべきテスト}
- {設定変更}
- {アーキテクチャ上の改善}
```

## 既存スキル・コマンドとの使い分け

| 観点 | requesting-code-review | differential-review | /review-respond |
| --- | --- | --- | --- |
| 目的 | 一般的なコードレビュー | セキュリティ特化レビュー | Copilotレビュー対応 |
| タイミング | 機能完了時・PR作成前 | セキュリティ変更時 | PR作成後 |
| フォーカス | 可読性・設計・ロジック | 脆弱性・攻撃経路・暗号 | レビューコメント対応 |

## 前提条件

- `git` コマンドが利用可能であること（追加の外部ツール不要）
