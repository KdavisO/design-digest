# Test-Driven Development

## トリガー

以下の場合にこのスキルを適用する:

- 新機能の実装（`feat:` コミット）
- バグ修正（`fix:` コミット）
- リファクタリングで振る舞いが変わる場合

## ワークフロー: RED-GREEN-REFACTOR

### 1. RED: 失敗するテストを書く

- 実装したい振る舞いを **テストとして先に記述** する
- テストを実行し、**失敗することを確認** する（テストランナーの出力を確認）
- 失敗理由が「未実装」や「期待値不一致」であることを確認する（環境エラーではないこと）

```
# NG: テストを書かずに実装を始める
# OK: テストを書く → 失敗を確認 → 実装に進む
```

### 2. GREEN: テストを通す最小限の実装

- 失敗しているテストを **通すために必要な最小限のコード** のみを書く
- 将来の要件やエッジケースへの対応はこの段階では行わない
- テストを実行し、**パスすることを確認** する

### 3. REFACTOR: コードを整理する

- テストがパスしている状態を維持しながらリファクタリングする
- リファクタリング後もテストがパスすることを確認する
- 新たなエッジケースが見つかったら、手順1に戻る

## テスト設計の原則

### 正常系・異常系・境界値を網羅する

CLAUDE.md のテスト設計規約に従い、以下を網羅する:

- **正常系**: 期待通りの入力での動作
- **異常系**: 不正入力、タイムアウト、権限エラー
- **境界値**: 空配列、空文字列、0、負数、null/undefined

### アンチパターン

- **実装の内部構造をテストしない**: モックは外部インターフェースの振る舞いに基づいて設計する
- **テストを通すためだけのコードを書かない**: テストが実際の要件を反映していることを確認する
- **複数の振る舞いを1つのテストに詰め込まない**: 1テスト1アサーション（関連するアサーションのグループは可）
- **テスト間に順序依存を作らない**: 各テストは独立して実行できること

## `.claude/rules/git-conventions.md` との関係

- TDD で書いたテストは、セルフレビューチェックリストの「テスト十分性」項目を満たす根拠となる
- TDD サイクルの各段階でコミットする必要はない（GREEN 後にまとめてコミットで可）
- コミット時のセルフレビューは従来通り必須
